# Task Runner v1.0.3 Release Notes

**Release Date:** 18 January 2026
**Version:** 1.0.3  
**Status:** Enhancement Release  

## ğŸ‰ Overview

Task Runner v1.0.3 introduces support for orchestrator tasks - tasks with no commands that serve to group and coordinate dependencies. This enhancement allows you to create cleaner workflow definitions by having tasks that exist purely to orchestrate other tasks.

## âœ¨ Enhancements

### Orchestrator Tasks Support
- **Added**: Tasks can now have empty commands if they have dependencies
- **Impact**: Enables cleaner workflow organization by creating tasks that exist purely to coordinate other tasks
- **Use Case**: Perfect for creating high-level task groupings like `check`, `deploy`, or `dev` that run multiple related tasks
- **Implementation**: 
  - Validation now allows tasks with empty `commands` array if they have `dependencies`
  - Tasks with empty commands and no dependencies will still fail validation (must have at least one or the other)
  - Execution handles orchestrator tasks gracefully - they complete successfully once all dependencies finish

### Validation Improvements
- **Enhanced**: Task validation now recognizes orchestrator tasks as valid
- **Impact**: Users can now define tasks that serve as workflow orchestrators without needing placeholder commands
- **Example**: A `check` task can now have `"commands": []` and `"dependencies": ["lint", "format", "test"]` to run all checks

## ğŸ“¦ What's Included

This release includes:
- Support for orchestrator tasks (tasks with dependencies but no commands)
- Improved task validation to allow empty commands for dependency-only tasks
- Enhanced workflow organization capabilities
- All features and functionality from v1.0.2 remain unchanged

## ğŸ’¡ Example Usage

### Orchestrator Task Example

```json
{
  "tasks": {
    "lint": {
      "description": "Run linting",
      "commands": ["npm run lint"]
    },
    "format": {
      "description": "Format code",
      "commands": ["npm run format"]
    },
    "test": {
      "description": "Run tests",
      "commands": ["npm test"]
    },
    "check": {
      "description": "Run all checks",
      "dependencies": ["lint", "format", "test"],
      "commands": []
    }
  }
}
```

Running `task-runner run check` will execute `lint`, `format`, and `test` in dependency order, and then the `check` task itself will complete successfully (no commands to run).

### Complex Orchestration Example

```yaml
tasks:
  build:backend:
    description: Build backend
    commands:
      - "cargo build"
    working_dir: "./backend"
    
  build:frontend:
    description: Build frontend
    commands:
      - "npm run build"
    working_dir: "./frontend"
    
  test:backend:
    description: Test backend
    commands:
      - "cargo test"
    working_dir: "./backend"
    dependencies: ["build:backend"]
    
  test:frontend:
    description: Test frontend
    commands:
      - "npm test"
    working_dir: "./frontend"
    dependencies: ["build:frontend"]
    
  build:
    description: Build everything
    dependencies: ["build:backend", "build:frontend"]
    commands: []
    
  test:
    description: Test everything
    dependencies: ["test:backend", "test:frontend"]
    commands: []
    
  ci:
    description: Run CI pipeline
    dependencies: ["build", "test"]
    commands: []
```

This allows you to run `task-runner run ci` to execute the entire CI pipeline, or `task-runner run build` to build everything, without needing to remember all the individual task names.
